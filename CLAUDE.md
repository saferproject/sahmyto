## Project Briefing

**سهمیتو (Sahmito)** is a Persian-language (RTL) web app for **income and expense management for transportation businesses across all vehicle types** — a fleet finance ledger. The central domain concept is a **karboom (کاربوم)**: a per-context ledger that aggregates **incomes, expenses, partners, drivers, and third-party insurance**. Authors: Amir Allahdadian, Michaeil Nobakht.

The app is mobile-first and entirely client-rendered (almost every component is `"use client"`). It talks to a separate backend API (Laravel-style, see env URLs) — there is no server-side data layer in this repo.

Primary user flows by route segment:
- `/` — landing with Intro/Login buttons
- `/intro` — onboarding carousel
- `/login` → `/login/verify` — phone-number + OTP auth
- `/dashboard` — authenticated app shell (header, footer, guard)
- `/dashboard/karbooms/*` — the core feature: `incomes-list`, `expenses-list`, `partners-list`, `drivers-list`, `third-party-insurance`
- `/dashboard/profile` (+ `profile-picture`), `/dashboard/tutorial`

## Commands

Package manager is **Bun** (`bun.lock`; `package.json` uses bun-specific `trustedDependencies`/`ignoreScripts`).

```bash
bun install        # install deps
bun dev            # dev server (next dev) on http://localhost:3000
bun run build      # production build (next build)
bun start          # serve production build
bun run lint       # eslint
```

There is **no test setup** in this project — no test runner, scripts, or test files exist.

## Architecture

### Stack
Next.js 16 (App Router) · React 19 with **React Compiler enabled** (`reactCompiler: true`, `babel-plugin-react-compiler`) · TypeScript (strict) · MUI v7 + Emotion · Tailwind CSS v4 · TanStack React Query · Zustand · react-hook-form + zod · dayjs + jalaliday · **iconsax-reactjs** (the single icon library — use `variant="Linear|Bold|Broken|…"`, not a `weight` prop).

Path alias: `@/*` → repo root (e.g. `@/app/proxy`).

### Folder convention (feature co-location)
Code is organized by route segment, and within each segment by **underscore-prefixed category folders**: `_components`, `_hooks`, `_interfaces`, `_types`, `_schemas`, `_services`, `_stores`, `_providers`, `_constants`, `_utilities`, `_assets`, `_layouts`. App-wide shared versions of these live at `app/_*`. When adding a feature, follow the nearest sibling segment's structure rather than inventing a new layout.

### Data fetching & auth
**`app/proxy.ts` → `fetchWithAuth<T>(path, options)`** is the single fetch client. It prefixes `NEXT_PUBLIC_API_URL`, attaches `Authorization: Bearer <token>` from `localStorage`, and on **401 it clears localStorage and redirects to `/login`**. All `_services/*` use this, and new API calls should go through it. All responses are wrapped in `BaseResponse<T>` (`app/_interfaces/base-response.ts`).

### The service → hook → component data pattern
1. `_services/<name>-service.ts` — plain object of functions calling `fetchWithAuth`.
2. `_hooks/use-...-endpoint.ts` — wraps the service in `useQuery`/`useMutation`.
3. Component consumes the hook.

User-facing feedback is **centralized** in `app/query-client.tsx` (`ReactQueryProvider`): the `QueryCache`/`MutationCache` automatically fire notistack snackbars (Persian text) on query error, mutation success, and mutation error. Individual hooks generally should **not** add their own toast on success/error — it's handled globally. Per-mutation side effects (closing a drawer, navigation, form `reset`) are passed as the **second arg to `mutate(vars, { onSuccess })`**, not via a `useEffect(isSuccess)`. Lists wrap their rendering in `app/_components/query-state.tsx` (`<QueryState isLoading isError isEmpty>`) for consistent loading/error/empty UI. Query defaults: `staleTime: 60s`, `retry: 0`, `refetchOnWindowFocus: false`.

### State management
- **Server state**: React Query.
- **Client state**: Zustand using the **vanilla store + Context provider + selector hook** pattern (see `app/_stores/user-info-store.ts`, `app/_providers/user-info-provider.tsx`). Each store is created per-provider-instance and consumed through a typed selector hook that throws if used outside its provider. This trio is generated by the **`createStoreContext` factory** (`app/_utilities/create-store-context.tsx`) — pass the vanilla `createStore` fn + a hook name, get back `[Provider, useStore]`; don't hand-roll new ones. `user-info-store` also persists the user to `localStorage`.

### Provider composition (per-layout, not in root layout)
The root `app/layout.tsx` only sets `<html lang="fa" dir="rtl">` and the font. Providers are composed inside each route group's layout (`app/login/layout.tsx`, `app/dashboard/layout.tsx`). Dashboard order:
`ThemeRegistry → ConfirmationDialogStoreProvider → UserInfoStoreProvider → SnackbarProvider → ReactQueryProvider → AuthenticationGuard`.
**Order matters:** `ReactQueryProvider` calls `useSnackbar()`, so it must be nested inside `SnackbarProvider`. Auth gating for the dashboard is `app/dashboard/_utilities/authentication-guard`, which renders a loader until the profile query confirms the session (and redirects to `/login` when no token/user is in `localStorage`).

### Styling, RTL & localization
- `ThemeRegistry` (`app/theme-registry.tsx`) wires Emotion with **RTL** (`@mui/stylis-plugin-rtl` + `prefixer`), the MUI theme (`app/theme.ts`), and `LocalizationProvider` with the **Jalali (Persian) calendar** — dayjs is globally extended with `jalaliday` and set to the `fa` locale here.
- MUI theme: `direction: "rtl"`, primary orange `#ff8500`, rounded shapes; date pickers use `faIR` locale with custom Persian button labels.
- Tailwind v4 is configured **in CSS** via `@import "tailwindcss"` and an `@theme` block in `app/globals.css` (custom `primary`/`secondary`/`heading`/`body` color tokens in oklch). A global `* { font-family: var(--font-yekan-bakh) !important }` rule forces the local **YekanBakh** font (includes Persian numerals) everywhere; scrollbars are hidden globally.
- Use Jalali dates via dayjs throughout; form date fields are typed as `Dayjs` in zod schemas (`z.custom<Dayjs>()`).

### Forms
react-hook-form + `zodResolver`. Convention: schema in `_schemas/*-form-schema.ts` (exports the schema as default plus an inferred `*FormType`), and a `_hooks/use-*-form.ts` that returns `useForm({ resolver, defaultValues })`. Validation messages are Persian strings.

### Config notes
- `next.config.ts`: `reactCompiler: true`, `allowedDevOrigins` for LAN dev, and `images.remotePatterns` **derived at build time from `NEXT_PUBLIC_IMAGE_ASSETS_URL`** (so `next/image` hosts never drift from the env). Commented `output: 'export'` hints at a possible static-export target.
- Env: `NEXT_PUBLIC_API_URL` and `NEXT_PUBLIC_IMAGE_ASSETS_URL` (see `.env.development` / `.env.production*`). All API/image hosts come from these.
- `middleware.ts` is currently a no-op pass-through.
- Prettier uses `prettier-plugin-tailwindcss`; the workspace sets `printWidth: 140` (`.vscode/settings.json`).

### Gotchas
- The `app/dashboard/karbooms/_assets/_Vectors/` folder has a capital `V` (other asset folders use lowercase `_vectors`/`_images`) — match it exactly when importing.
- The repo has **pre-existing TypeScript and ESLint errors** in files around the expense/income drawers and detail layouts (e.g. `Expense` lacks `started_at`/`ended_at`, several `@ts-ignore`s, refs accessed during render). `next build` will fail on them until they're addressed — they predate the current refactors, so don't assume your change caused a red build; verify with `bunx tsc --noEmit`.

## Design Review (ongoing)

A part-by-part pass over the UI to find bugs and improve cross-device (mobile) stability. **Don't add new design** — fix bugs and make existing design stable/consistent. Workflow: review a part → report findings (bugs + cross-device) → user picks what to fix → fix one by one.

### Design-check dimensions (apply to every part)
Layout & spacing · RTL correctness (direction, icon mirroring, start/end not left/right) · responsiveness (mobile-first, overflow, safe areas, keyboard) · typography & color (theme tokens, primary `#ff8500`, contrast) · component consistency (MUI, rounded shapes, iconsax `variant`) · states (loading/skeleton, empty, error, disabled) · Persian content (labels, numerals, Jalali dates) · interaction (touch targets, drawers, snackbars, dialogs).

**RTL number rule:** Persian numerals render LTR, but numeric inputs (amounts, phone, OTP, plate, dates) must stay **right-aligned** — don't switch the field to `dir="ltr"`/left-align.
